Main Argument

Exposing an analytics platform’s APIs as tools callable by LLM-based AI agents (function calling) lets analysts run whole analytical workflows in natural language — a concrete step toward autonomous data analytics.

Problem / Context

Data science is complex and demands scarce human expertise, so automating its software-engineering tasks is valuable. But traditional form- and API-based analytics platforms impose real cognitive overhead and technical barriers — they require users to formulate code-like requests and know configuration parameters, which non-technical users cannot do, bottlenecking insight on a small pool of specialists. Existing GenAI approaches mostly integrate AI as copilots or address only a single stage of the analytics lifecycle. The paper’s differentiators: it (1) considers the whole analytics lifecycle, (2) exposes platform functionality as tools accessible to external systems, and (3) gives a systematic framework to design agentic analytics workflows — converting an API-based platform into a conversational, natural-language interface driven by multiple LLM agents.

Key concepts

Function calling / tool use
Giving an LLM structured “tools” (function name + JSON input parameters) it can invoke to fetch data or run analyses — what turns a chatbot into an agent that actually executes workflows.
Model Context Protocol (MCP)
Anthropic’s JSON-based standard (host = LLM/agent, client = adapter, server = wrapped tools) used to wrap the platform’s function-calling API as reusable tools, boosting interoperability.
AI agents
LLMs given a name, personality/description, tools and expected output; multiple agents converse and cooperate through a defined process.
The “Flow” process
P = ⟨States, Agents, Tasks, Transitions⟩ — here 6 states and 5 agents (User Input → Manager → Coder / Analyst / Result Expert → Fulfilment), with self-reflect and human-in-the-loop mechanisms.
Enablers
Prompt engineering, knowledge injection (RAG), memory expansion, function calling, fine-tuning, and AI agents — the techniques that adapt LLMs to the task.
LLM-as-a-judge
Using a stronger model (Deepseek-r1) to score outputs on defined metrics — scalable, explainable evaluation via the Deepeval framework’s G-EVAL methodology.

Data & Method

The authors extend an Analytics-as-a-Service (A4A) platform: its GraphQL/API functions are wrapped as function-calling tools exposed through an MCP server. A coordination process (the “Flow”) mediates between the data analyst, the AI agents and the platform, producing a chat-based interface. The industrial use case is EV Li-Ion battery testing (chamber/tester sensor data, machine statuses), with five descriptive analyses + one predictive analysis (chamber-status forecast).

  1. Tool exposure — platform functions → function-calling API → MCP-wrapped tools (Result Attributes, Build Analysis, Algorithm Code/Description, Intents, Results, Prediction, etc.).
  2. Flow architecture — 6 states / 5 agents: User Input → Manager (routes) → Coder / Analyst / Result Expert (tool-using) → Fulfilment (checks answer, loops back if insufficient); self-reflect + human-in-the-loop.
  3. Benchmark experiments — 3 architectures (LLM, Agent, Flow) × 2 open-source models (Qwen 2.5:32b, Mistral-small:24b) × 16 cases × 3 seeds = 288 experiments; hosted via Ollama on 2×RTX 4090 (48GB); judged by Deepseek-r1 on 5 Deepeval metrics.
  4. User study — N = 11 (highly educated, technically experienced); each user does 3 tasks with both the form-based and the Flow+Qwen chat interface; logged task time, #actions, plus Likert usability/satisfaction/confidence; significance via Mann-Whitney U.

Key Results

In the benchmark, the multi-agent Flow architecture with Qwen scored best overall (0.86), a +7.2% mean improvement over other approaches, and led on Correctness and Tool Correctness.

The chat interface was better by at least one point on usability, ease of use, satisfaction and timeliness, and notably reduced the “familiarity barrier” — users no longer needed to already know the platform or the use case. Despite the known hallucination risk, participants reported high confidence and an absence of hallucinations, and valued that the agents’ executed queries were displayed (explainability/trust). Trade-off: the Flow architecture is slower per run (~2–3× longer mean execution time) than the single-LLM baseline.

5 Lessons Learned

  1. Natural language is emerging as the next BI interface — it collapses the cost of learning a platform’s tools and menus.
  2. Function calling / tool use is precisely what turns a general chatbot into an agent that actually executes analytics workflows.
  3. The measurable win is time and effort (roughly 2.5× faster, about 4× fewer actions), not just subjective user delight.
  4. GenAI democratises analytics — non-experts can now run workflows that used to require specialist platform knowledge.
  5. Risks remain real: hallucination and trust must be explicitly measured, not simply assumed away by a friendlier interface.

Limitations & critique

  • Small user study: only N = 11 participants, mostly highly-educated technical experts — not representative of the non-technical users the system aims to empower.
  • One platform, open-source models: evaluated on a single analytics platform (A4A) with quantised open-source LLMs, which are weaker than frontier models; results may not transfer.
  • Speed/robustness trade-off: the best-scoring Flow architecture is markedly slower (2–3× longer execution) and showed performance instability/outliers — sophistication costs latency.
  • Limited task set: 16 benchmark cases and a battery-testing use case; broader analytical workflows are untested.
  • Hallucination not solved: although low here, hallucination and trust remain fundamental unresolved LLM risks that the paper measures but cannot eliminate; the work is described as “preliminary.”

Exam focus

Exam angle: “What does ‘autonomous data analytics’ actually mean, and what evidence supports it?” Answer: platform APIs exposed as LLM function-calling tools (via MCP) let a multi-agent “Flow” chat interface replace form-based analytics; a user study (N=11) shows task time drops 6.71→2.70 min and actions drop 14.48→3.30, usability 3.09→4.64, with the familiarity barrier reduced — while hallucination risk is explicitly measured, not ignored. Note the speed/robustness trade-off of the Flow architecture.
