Ch.7 · Deep Learning & Cognitive Computing

Key idea · Deep learning's superiority = automatic feature extraction & representation — it learns the features itself, where classic ML needs hand-engineered ones. "Deep" = more hidden layers.

Placement — AI ⊃ Machine Learning ⊃ Representation Learning ⊃ Deep Learning. Classic ML = manual feature engineering → algorithm; DL = raw input → learned features → output.

Shallow ANN (Artificial Neural Network) — abstraction of the brain; neuron = PE (Processing Element). Multi-input neuron: n = w₁p₁+w₂p₂+…+w_RP_R + b = wp+b, then a = f(n). Elements (Tech Insight 7.1): PE · network structure · hidden layer(s) · input · output · connection weights · summation function (combine incoming signals into one value) · transfer/activation function (translate input signal → output). Common transfer functions: sigmoid, tanh, ReLU, linear, step.

Learning process (3 steps) — (1) compute temporary outputs → (2) compare with desired targets → (3) adjust weights & repeat. Backpropagation (5 steps): initialise weights randomly → read input vector + desired output → compute actual output → compute the error → change weights working backward.

Black box — ANNs are opaque. Fix = sensitivity analysis based on leave-one-out (drop each input, measure the drop in performance → variable importance). Case 7.2: sensitivity analysis reveals injury-severity factors in traffic accidents.

Deep neural networks — more hidden layers; needs GPU (+ CUDA by NVIDIA) not just CPU; needs large datasets; inputs are tensors (N-dimensional arrays — an image = 3-D tensor). Deeper layers ≠ simply more neurons; random weights & pretraining materially change results.

CNN — most popular MLP-based DL method. Convolution layer applies a kernel/filter over the input matrix (eg a 2×2 kernel over a 3×6 input); pooling consolidates large tensors into smaller ones — fewer parameters, keeps only the important features. Architectures: AlexNet (ImageNet), GoogLeNet (Inception). Apps: Google Lens, face recognition. CNN for text: Google word2vec → word embeddings (words as vectors); used for relation extraction.

ChatGPT / LLMs — latest large language models (OpenAI); fastest AI adoption to date. Apps: customer service, healthcare, marketing, programming. Limitations: biases, false answers (hallucination), obsolete data.

Frameworks — TensorFlow (Google; TPUs) · Keras (high-level API) · Torch · Caffe/Caffe2 (Facebook) · Theano (U. Montreal).

Vignette · Danske Bank — fighting fraud with DL/AI: old rules-based detection replaced by deep learning → higher accuracy & better ROC; key managerial point = the false-positive vs false-negative (Type I vs Type II) trade-off. Fraud types: phishing, identity theft, ATM skimming, loan fraud. Case 7.5 (ED bounce-backs): adding physician notes lifts Accuracy 0.668→0.748, Recall 0.615→0.756, AUROC 0.674→0.766.

Cognitive computing — systems using mathematical models to emulate the human cognition process for complex problems where answers can be imprecise. Four properties: adaptive · interactive · iterative & stateful · contextual. Built from data mining + pattern recognition + deep learning + NLP. Flagship: IBM Watson (Jeopardy!). Cognitive search = handles many data types, contextualises the search space, uses AI.

Ch.8 · Prescriptive Analytics — Optimization & Simulation

Key idea · Descriptive + Predictive → Prescriptive. Descriptive/predictive create the alternatives; prescriptive picks the best decision — by optimization (guarantees the optimum of the modeled problem) or simulation (only "good enough", by experiment).

Model components — decision variables (what you control: how much to produce, staffing, ad budget) · uncontrollable variables/parameters (environment: inflation, competition, machine capacity) · result variables (outcomes: profit, cost, market share). Linked by equations/inequalities: P = R − C; PV P = F/(1+i)ⁿ → 100,000/(1.1)⁵ = 62,092.

Certainty / risk / uncertainty — Certainty: all outcomes known → may yield the optimum. Risk: several outcomes with known probabilities → compute expected values. Uncertainty: outcomes known, probabilities unknown → optimistic/pessimistic approaches.

LP (Linear Programming) — optimal allocation of scarce resources among competing activities; all relationships linear. 5 characteristics: limited resources · used to make products/services · 2+ ways to use them · each activity returns value toward the goal · allocation restricted by constraints. Steps: identify decision variables, objective function + coefficients, constraints → represent (LINDO formulation / Excel cells) → run & observe.

MBI product-mix LP (know this cold) · X₁ = CC-7 units, X₂ = CC-8 units. Max Z = 8000X₁ + 12000X₂ s.t. Labor 300X₁+500X₂ ≤ 200,000; Materials 10,000X₁+15,000X₂ ≤ 8,000,000; X₁ ≥ 100; X₂ ≥ 200. Solution: X₁ = 333⅓, X₂ = 200 → profit ≈ $5,066,667/month (labor is the binding constraint).

Decision table — investment eg · Bonds 12/6/3 · Stocks 15/3/−2 · CDs 6.5/6.5/6.5 (solid growth / stagnation / inflation). Uncertainty: optimistic → stocks (15); pessimistic → CDs (6.5). Risk (p = .5/.3/.2): Bonds = 8.4% ✔ · Stocks = 8.0% · CDs = 6.5% → bonds by expected value. Multiple goals (yield/safety/liquidity) can flip the answer.

Multiple goals — handled by utility theory, goal programming, expressing goals as LP constraints, a points system. Decision trees — graphical; induced from data (DM) or from experts (knowledge-driven); cumbersome with many alternatives.

Simulation — the "appearance" of reality; a technique for experiments on a model to study dynamic behaviour; used when the system is too complex for other techniques. Descriptive, NOT normative — cannot guarantee an optimum. Use only when numerical optimization is not possible. Advantages: simple theory, time compression, try alternatives, manager's perspective, handles real complexity, only tool for unstructured problems. Disadvantages: no optimal solution, costly/slow to build, problem-specific (doesn't transfer), easy to over-sell, needs skills.

Methodology (7): define problem → construct model → test & validate → design experiments → conduct experiments → evaluate results → implement. Types: stochastic (random inputs) vs deterministic · time-dependent vs independent · Monte Carlo (X = A + B, all probability distributions) · discrete-event vs continuous vs agent-based. VIS/VIM = visual interactive simulation (graphics + animation, RFID/GIS, VR). Tools: Arena, Simio, Crystal Ball, Palisade.

Supermarket simulation · Objective: least-cost mix of clerks & baggers keeping the queue under a limit 95% of the time. Decision vars = # counters, # baggers. State vars = # shoppers in store / per counter. Data = arrival rates, shopping-time distribution, checkout stats.

Expert Systems (rule-based AI)

Key idea · An ES imitates an expert's reasoning in a narrow domain. Its signature = knowledge (rules) SEPARATED from reasoning (inference engine) + it can explain itself. ES augment, not replace experts (non-expert:expert ≈ 100:1).

Expert = high proficiency in judgments in a specific domain. Expertise (4): extensive domain knowledge · heuristic rules · meta-knowledge/meta-cognition · compiled behaviour. Transferring expertise: acquisition → representation → inferencing → transfer (expert → computer → non-expert). Also: symbolic reasoning, deep knowledge, self-knowledge.

Conventional programExpert system
Knowledge + control mixedKnowledge base separate from inference
No explanationExplains why & how
Needs complete/exact dataCopes with incomplete/uncertain data
Changing logic = reprogrammingChange the rules — easy
Numeric, algorithmicSymbolic, heuristic

Architecture (7 components) — knowledge acquisition subsystem (via the knowledge engineer) · knowledge base (long-term memory: facts + rules) · inference engine (the brain: chains & fires rules) · user interface (Q&A dialogue) · blackboard/workplace (short-term working memory) · explanation subsystem (justifier) · knowledge-refining system. Two environments: development (expert + knowledge engineer) vs consultation (the end user).

Knowledge engineering (KE) — acquire knowledge from experts and convert it into the knowledge base; goals = help experts articulate what they do + document it reusably. 5 stages + feedback loop: knowledge acquisition → representation → validation → inferencing → explanation & justification (raw → codified → validated knowledge). Difficulty: experts can't articulate tacit knowledge, disagree, are expensive. Validation = performance (right system?) · Verification = built right (matches specs) · Evaluation = overall value.

Representation = production rules — condition–action pairs IF … THEN … ELSE. Forms: IF premise THEN conclusion · conclusion IF premise · with ELSE. Knowledge (declarative) rules = facts/relationships → stored in the knowledge base. Inference (procedural) rules = how to solve, incl. meta-rules → part of the inference engine (eg "IF data unknown THEN ask the user"; "IF >1 rule applies THEN fire the highest priority").

Forward chainingBackward chaining
Driven byData (facts → conclusions)Goal (goal → needed facts)
HowPremises match → assert conclusionStart at the action clause, work backward to verifiable conditions
UseMonitoring, planning, configurationDiagnosis (MYCIN-style)

Firing a rule = all its "IF" parts are satisfied; the engine checks rules until no more can fire or the goal is reached. Explanation: WHY (why is this fact requested?) vs HOW (how was the conclusion reached — the chain of fired rules). Purposes: intelligibility, uncover knowledge-base gaps, training/evaluation, user trust.

Classics — DENDRAL (molecular structure) · MYCIN (diagnose/treat bacterial infections) · XCON (optimal computer configuration). 10 problem areas: interpretation · prediction · diagnosis · repair · design · planning · monitoring · debugging · instruction · control. Development: scope → identify experts → acquire knowledge → pick tools (shell, eg Exsys, vs custom) → code → evaluate & launch.

Rule example (inventory) · IF current_stock < reorder_point AND lead_time ≤ 3d AND season = "peak" THEN order 150% of standard · IF …lead_time > 7d THEN expedited shipping, double quantity · IF days_to_expiry < 5 AND product = dairy THEN discount 30%. Loan-approval ES: IF score ≥ 700 AND DTI < 40% THEN approve · IF score < 550 THEN reject · IF 550–700 AND collateral THEN refer → backward chaining + justifier for fair-lending accountability.

Big Data Analytics

Key idea · "Big Data" is a misnomer — more than just big. And by itself it is worthless: Big Data + "big" analytics = VALUE. Hadoop coexists with (does not replace) the data warehouse.

The Vs — Volume (size) · Variety (structured + unstructured; schema-on-read) · Velocity (speed → stream analytics) · Veracity (quality/trust) · Variability (flow & meaning change) · Value. Scale: LHC 1 PB/s · Boeing jet 20 TB/hr · Facebook 500 TB/day · YouTube 1 TB/4 min. kB 10³ → MB 10⁶ → GB 10⁹ → TB 10¹² → PB 10¹⁵ → EB 10¹⁸ → ZB 10²¹ → YB 10²⁴.

7 critical success factors — (1) a clear business need (aligned to vision/strategy) · (2) strong, committed sponsorship (executive champion) · (3) alignment of business & IT strategy · (4) a fact-based decision culture · (5) a strong data infrastructure · (6) the right analytics tools · (7) the right people/skills.

4 enablers — in-memory analytics (whole dataset in RAM) · in-database analytics (procedures next to the data) · grid computing & MPP (massively parallel) · appliances (HW+SW+storage in one unit). 6 challenges — data volume · data integration · processing (stream) · data governance (security/privacy/access) · skill availability (data scientist) · solution cost (ROI).

MapReduce — Google's model that distributes processing of huge multi-structured files across a cluster of ordinary machines. Map = split & process in parallel (key–value pairs); Reduce = aggregate into the final result. MapReduce provides CONTROL for analytics, not the analytics itself.

Hadoop — open-source framework to store & analyse distributed, unstructured data on cheap commodity hardware (scale out). Created by Doug Cutting at Yahoo! → Apache. Components: HDFS (a file system, NOT a DBMS) · Name Node (primary facilitator) · Secondary Node (backup) · Job Tracker · Slave Nodes. Ecosystem: Hive (SQL-like DW — resembles SQL but isn't standard SQL) · Pig · HBase/Cassandra (NoSQL = "not only SQL") · Flume · Oozie · Ambari · Avro · Mahout · Sqoop.

Demystifying facts — Hadoop = an ecosystem, not one product; open source but also sold by vendors; HDFS ≠ DBMS; Hive ≠ standard SQL; Hadoop ≠ MapReduce; MapReduce = control, not analytics; Hadoop is about data diversity, not just volume.

Coexistence (5 patterns) — Hadoop as (1) store/archive multi-structured data, (2) filter/transform/consolidate (the "refinery"), (3) analyse & publish results to the DW; (4) an RDBMS with MapReduce as an investigative platform; (5) a front-end query tool over both. Use cases: Hadoop = repository & refinery + active archive; DW = performance, integrated business value, interactive BI.

Stream analytics — data-in-motion / real-time analytics: extract actionable information from continuously flowing data (the Velocity V). Why: storing it all may be infeasible, or the data loses value if analysed later. Related: perpetual analytics, critical event processing. Energy use case: sensor + meteorological + smart-meter data → streaming analytics (predict usage, production, anomalies) → capacity and pricing decisions. Sectors: e-commerce, telecom, cyber security, power, finance, health, government.

Cases · Telecom churn (opening vignette — multi-source, high volume; sessionizing = stitching raw events into user sessions) · investment bank/MarkLogic (single source of truth: 1 DB replaced ~20 copies) · eBay · flu nowcasting from social media · EMR comorbidity networks (urban vs rural). Vendors: Cloudera, MapR, Hortonworks, IBM InfoSphere BigInsights, Teradata Aster.

Ch.10 · New AI-Based Trends in Analytics

Key idea · Analytics is reshaped by where compute lives (cloud), what counts as data (location, images, sensors = "alternative data"), and what is connected (IoT). IoT is NOT an application — it is the infrastructure/platform that feeds analytics.

Cloud computing — "a model enabling convenient, on-demand network access to a shared pool of configurable computing resources (networks, servers, storage, services) rapidly provisioned and released with minimal management effort." A new name for older trends (utility/grid/on-demand computing, ASP, SaaS). Service-oriented DSS = cloud analytics; 4 componentes as a service: operational systems · data warehouses · OLAP · end-user components.

Virtualization (essential enabler) — network (split bandwidth into channels) · storage (pool many devices into one) · server (mask physical servers → centralise management). Deployment: private (one org) · public (shared over Internet) · hybrid. Top-3 security threats (Cloud Security Alliance): data loss & leakage · hardware failure · insecure interface. Other challenges: service level, data governance, loss of control, privacy, legal liability, cross-border politics.

Location-based analytics — adds "where" to who/what/when/how-much. Geospatial (static, from GIS; leader ESRI/ArcGIS) → crime analysis, disease spread, agriculture. Real-time location intelligence (dynamic) — GPS + Wi-Fi + cell-tower triangulation; movement patterns → promotions/pricing; augmented reality is an extension. Consumer apps: Waze (crowd-sourced hazards), ParkPGH (predicts parking). Key concern: loss of privacy. Image/alternative data — satellite imagery: World Bank (urban planning), EarthCast (60 satellites → airline weather), Imazon (Amazon deforestation), Global Fishing Watch (illegal fishing).

IoT — a network connecting many objects, each with an embedded microprocessor, wirelessly, anytime/anyplace. Working definition: connected devices, each with a unique identifier (UID), collecting and transferring data automatically. 5 characteristics (Miller): ① large numbers of things connected · ② each has a unique ID/IP · ③ each can receive, send & store data automatically · ④ mostly over the wireless Internet · ⑤ built on M2M communication. Internet connects people; IoT connects things. Kevin Ashton coined the term ("…computers can understand the world around them and become humanity's nervous system").

How IoT works (the chain) · things → sensors collect data → displayed/stored/analysed (data mining) → information becomes knowledge/intelligence → expert systems or ML turn it into decision support → improved actions & results → new use cases. 4 building blocks: hardware (devices/sensors/actuators) → connectivity (hubs, gateways, Internet/cloud) → software backend (logic, usually cloud) → applications. Platforms: AWS IoT · Azure IoT Suite · GE Predix · IBM Watson IoT.

Sensors & RFID — a sensor automatically collects data about events in its environment; has distance & precision limits; very-short-range = proximity sensors (more reliable). RFID = radio-frequency waves to identify objects; same family as barcodes/magnetic strips but stores richer data; works with tags + readers; active (long-range) vs passive (short-range) tags; spread via retail supply chains.

IoT benefits — automate processes (cost ↓), productivity ↑, new revenue streams, optimise asset utilisation, sustainability, anticipate needs, identify problems before they occur, cheap tracking. Drivers — more things online, autonomous things, broadband spread, falling device/sensor & connection costs, smartphone penetration, wearables.

Applications — SNCF (French rail): 15,000 trains, 30,000 km, ~14 M passengers, IBM Watson behind it. Smart home: interconnected lights/appliances/security → comfort, security, low energy cost. Barriers = the 3 C's: Compatibility (too many vendors), Communication (consumers disagree what a smart home is), Concentration (target the interested segment, e.g. Gen Y) + cost, privacy, security, ease of use. Smart digital factory: self-optimise · self-adapt/learn in real time · autonomously run production. Smart city: transportation is the key area (Valerann smart road studs; HK Smart Mobility Consortium).

Autonomous vehicles — Google started it, Tesla made it famous. Issues: cost of real-time 3D maps · AI software nimbleness · readying automakers & insurers · sensor cost/quality · IoT connectivity. 5G (5 benefits): ① very high speed even in dense areas · ② connects everything (massive IoT + M2M) · ③ real-time latency · ④ better security & privacy · ⑤ service-based architecture (SBA).

Other trends — RPA (Robotic Process Automation): software "bots" replicate human actions on repetitive, high-volume steps → productivity ↑, errors eliminated, cost ↓. Network science: analyses interactions among features/observations (e.g. comorbidity networks → predict hospital length of stay); spans all three echelons (descriptive, predictive, prescriptive). Web 3.0 (decentralized: blockchain, crypto, NFT) · metaverse & digital twins (a digital counterpart of a real product/system) · GPT/ChatGPT, LaMDA, BlenderBot.

Cases · Siemens (opening vignette): sensor data predicts train failures — the classic "1,000 sensors × 1 KB/s" data-volume question (≈ 2.6 TB/month). Discover Foods: IoT + ML keep food temperature stable (quality degrades when temperature fluctuates). Zion China: Azure IoT + Stream Analytics + ML + Power BI for mobile health. Ch.11 · Ethical, Privacy & Managerial Considerations

Key idea · Just because something is DOABLE through technology does not make it appropriate, legal, or ethical. Legal, privacy and ethical issues are interconnected. The manager's job: deploy responsibly — liability, privacy, bias, and the human impact (jobs, skills, resistance).

Opening vignette (analytics-journey lessons) — business problem vs research problem; define goal, value & the right metrics; check data access & governance; prediction vs causation; bias (how far predictions are from truth) vs variance (how much they fluctuate with different training data); overfitting (low train error, high test error) vs underfitting (high error on both); retrain models; explain results in business metrics; realistic timelines; business buy-in; the talent gap.

Implementation — 3 major development problems (Shchutskaya & Zharovskikh): ① the development approach differs from ordinary IT · ② learning from data — data quality determines outcome quality · ③ no clear view of how insights are generated (opacity + drift). Plus connectivity/integration with existing IT and security (cloud + robots create new vulnerabilities). Leverage: customize the customer experience · increase engagement (chatbots) · detect anomalies. CSF: upgrade skills, build an AI-savvy workforce; the top issue is employees' fear of job loss.

Deployment strategy (manager's checklist) — when to start & how to prioritize; DIY vs partner vs outsource; justify the investment (ROI); overcome employee resistance; arrange people–robot teams; decide which decisions to fully automate; protect security & privacy; handle job loss & retraining. Top management must grasp tactical and strategic opportunities, redesign the organization, and shape the future of work — failure to respond is blamed on senior managers. KPMG digital-labor guidelines: priority areas · employee strategy · providers/partners · benefit realization.

Implementing IoT — issues: organizational alignment, interoperability, security, privacy, connecting data silos, management, connected customers. Industrial-IoT competitive-advantage strategy (5): ① specify business goals (+ROI) → ② express an analytic strategy → ③ evaluate need for edge analytics (analyse data where it is collected) → ④ select analytics solutions → ⑤ continuous improvement closes the loop. Rollout: business case → prototype → pilot in one unit → org-wide deployment.

Legal issues — mostly liability, mostly unanswered: who is liable for an intelligent machine's advice? if the firm goes bankrupt on AI advice? if a physician accepts a wrong computer diagnosis? for a self-driving car crash? Are developers/auditors jointly liable? Who owns the knowledge ChatGPT/Bard gathers? Can management force experts to contribute expertise (and how are they paid)? At present there are very few laws on intelligent technologies — most relate to safety. AI in law: document review, contract analysis, legal research, predicting case outcomes; open issue = ownership of training content.

Privacy definition (memorise) · the right to be left alone and the right to be free from unreasonable personal intrusions. Two court rules: ① the right of privacy is NOT absolute — it is balanced against the needs of society; ② the public's right to know is superior to the individual's right to privacy.

Where privacy leaks — virtual assistants (Alexa/Siri/Google listen — "your voice assistant is spying on you"; privacy vs convenience) · mobile (cell-tower handoffs, GPS, Wi-Fi hotspots) · IoT networks · profiling (Rapleaf: email → full profile; BlueCava: multi-device profiles; Reputation.com: email → SSN) · workplace analytics (Humanyze sensor badges track all movement; Barclays heat/motion sensors timed bankers at their desks — the union is watching) · surveillance (Delaware AI dashcams, Facebook face recognition, Epicenter microchip implants) · connected cars ("a privacy nightmare" — who owns the data: CarPlay? Android Auto?).

Ethics · some ethical issues are upsetting but NOT illegal. The Facebook News-Feed experiment: showed users different feeds and measured emotional reactions — judged unethical because it lacked informed consent (and because Facebook is so large); FB acknowledged the error. Machine ethics = moral behaviour of AI beings; roboethics = moral behaviour of the designers/builders/users of robots. Cautionary: Tay chatbot shut down; algorithms may become racist; a self-driving car may face whom to save/kill. 10 Commandments of Computer Ethics (don't harm people, interfere, snoop, steal, bear false witness, copy unpaid software, use others' resources, appropriate others' output; do consider social consequences and act with respect).

Responsible AI — 6 principles: ① Fairness ② Reliability & safety ③ Privacy & security ④ Inclusiveness ⑤ Transparency ⑥ Accountability. Bias cases: Amazon's scrapped AI recruiting tool (gender bias learned from history) · COMPAS (racial bias in sentencing risk scores) · BlenderBot 3.0. Why managers care: Netflix's recommenders are AI-built and not immune to bias; AI market $387 B (2022) → $1,394 B by 2029; Accenture: AI could raise profitability ~38% by 2035.

O'Neil — Weapons of Math Destruction · a model must be: ① transparent (an unintelligible model → unintended consequences) · ② built on clear quantifiable objectives · ③ self-correcting + regularly audited, with new inputs/outputs constantly reconsidered.

Impact on organizations — new units (analytics/BI/data-science/AI departments) and C-suite titles (Chief Data Officer, Chief AI Officer). On managers: less expertise needed for many decisions · faster decisions · less reliance on experts · power is redistributed — the more information & analysis capability a manager has, the more power · routine decisions automated.

Impact on jobs — agreed: AI creates new jobs, many need retraining, the nature of work changes; no consensus on when/how much. AI accelerates labor-market polarization — growth at the top and bottom, losses in the MIDDLE. Already taken: online marketers, financial analysts, diagnosticians, reporters, middle management, bookkeepers, cashiers, drivers. Low-risk: nurses, doctors, dentists, coaches, social workers. Jobs get redesigned either low-skill (to automate) or high-skill (for humans). Tips: digital capabilities must come BEFORE AI; ML isn't the answer to everything; don't leave AI solely to technology teams; portfolio approach; the biggest challenges are people and business processes; top-leadership support is required.

Dangers & the debate — Musk: "potentially more dangerous than nukes"; Gates: concerned about superintelligence; Hawking: "could spell the end of the human race." Utopia camp: better quality of life (e.g. 20% crime reduction) — but concedes the road is rocky (jobs) and change may be turbulent. Citizen data scientist = a business user NOT fully trained in the maths/statistics of data science, who uses visual, intuitive software tools to build models for their own business use → helps alleviate the data-scientist shortage. (Citizen science = the public volunteers to help conduct research.)

↔Ch.6R12 · Sentiment Analysis + CEEMDAN for Consumer-Group Preferences (Wang 2026)

  • Main argument: Online reviews hide consumer preferences that are unstructured, emotional and time-dependent — combining NLP with decision modelling learns stable, interpretable group preferences.
  • Problem/Context: hotels get thousands of Ctrip reviews; they are scattered, implicit, ambiguous and shift over time, so managers can't turn them into decisions.
  • Data & Method: 14,133 → 10,505 cleaned Ctrip hotel reviews. (1) BERT-Base-Chinese fine-grained aspect sentiment → (2) BERTopic extracts 20 topics → (3) CEEMDAN decomposes the time series (separates short-term noise from long-term trend) → (4) UTA preference model derives criteria weights, thresholds & value systems per consumer group.
  • Key results: BERT sentiment F1 = 0.83, beating LSTM (0.761) and Random Forest (0.568); yields interpretable per-group priority factors → manager decision support.
  • 5 lessons learned
    1. Fine-grained (aspect-level) sentiment gives far deeper insight than a simple positive/negative label.
    2. Preferences change over time, so a trend/noise decomposition (CEEMDAN) is needed for stable conclusions.
    3. Combining NLP with a decision model (UTA) buys interpretability that pure deep learning lacks.
    4. Transformer models (BERT) clearly outperform LSTM and classical ML on messy review text.
    5. Review mining becomes actionable: it guides service, marketing and resource decisions per customer segment.

↔Ch.7R13 · Brain-Tumour Classification with CNN + Soft-Voting Ensemble (Puspita 2025)

  • Main argument: No single CNN is best — a soft-voting ensemble of several CNNs raises brain-tumour classification accuracy above any individual model.
  • Problem/Context: glioma, meningioma and pituitary tumours are major global health issues; manual MRI reading is slow and error-prone, so early automated detection is needed.
  • Data & Method: 7,023 MRI images (512×512) across glioma / meningioma / pituitary / no-tumour. Four CNNs — VGG16, MobileNet, ResNet50, DenseNet121 — each tuned (Adam, lr 0.0001, kernel regulariser 0.001), then combined by soft voting (average the predicted probabilities, take the argmax).
  • Key results: ensemble accuracy 97.67%, Cohen's Kappa 0.9688 — beating the individual CNNs and prior published methods.
  • 5 lessons learned
    1. CNNs learn image features automatically — no hand-crafted feature engineering (Ch.7's core claim).
    2. Ensembles beat single models: soft voting averages probabilities and cancels individual model errors.
    3. Transfer learning from pretrained architectures works even on modest medical datasets.
    4. Accuracy alone is not enough — Cohen's Kappa guards against agreement by chance in multi-class problems.
    5. DL supports (does not replace) radiologists: it is a screening/decision-support aid.

↔Ch.5/7R14 · Comparing ML Algorithms for Health-Insurance Pricing (Bau & Md Hanif 2024)

  • Main argument: Insurance pricing should not be guesswork — ensemble regression (random forest) predicts premiums best, keeping pricing profitable and cost-effective.
  • Problem/Context: post-COVID risk and uncertainty rose; insurers must price health packages within a profitable threshold, and previously relied on consultants/intuition.
  • Data & Method: US Health Insurance dataset. Four regression models compared: multiple linear regression, ridge regression, XGBoost, random forest; evaluated on MAE, MSE, RMSE, R²; the winner is then tuned with GridSearchCV hyperparameter search; feature-importance and actual-vs-predicted plots for interpretation.
  • Key results: Random forest wins on all four metrics (R² ≈ 0.85; ~0.86 after tuning); XGBoost > linear/ridge; tuning improves 3 of 4 metrics (all but MAE). Ensembles > linear models.
  • 5 lessons learned
    1. Always compare several algorithms — don't assume one is best for your data.
    2. Ensemble methods (bagging → RF, boosting → XGBoost) outperform linear/ridge regression on non-linear pricing data.
    3. Use multiple evaluation metrics (MAE/MSE/RMSE/R²) — a model can win on one and lose on another.
    4. Hyperparameter tuning (GridSearchCV) gives real, but not universal, gains.
    5. Visualisation (feature importance, actual-vs-predicted) converts a model into a managerial pricing decision.

↔ Ch.8 · Prescriptive: optimization & simulation

↔Ch.8R15 · AquaCrop + Optimization for Wheat Irrigation (Morocco) (Kaissi 2026)

  • Main argument: Coupling a crop simulation model with optimization algorithms finds irrigation schedules that cut water use sharply while preserving yield — simulation alone or farmer practice cannot.
  • Problem/Context: water scarcity in semi-arid Morocco; agriculture uses ~70% of freshwater; farmers over-irrigate wheat by habit rather than by a model.
  • Data & Method: AquaCrop-OSPy (FAO crop simulation) coupled with (i) unidimensional minimize_scalar (bounded) and (ii) multidimensional L-BFGS-B. Decision variables = soil-water depletion thresholds; objective = irrigation water productivity. Two fields (stressed / normal) in the Haouz Plain, two seasons (2016/17, 2017/18). Calibrated on canopy cover, biomass, ETa. Three strategies compared: Real (farmer) · NET (constant threshold) · SMT (stage-specific thresholds).
  • Key results: calibration R² 74–93%, NSE 0.71–0.95. Optimised SMT cuts irrigation 26–31% (Field 1) and 42–45% (Field 2) vs farmers' practice, sustaining yields 6.43 / 6.24 t/ha and raising water productivity to 2.08–2.33 kg/m³. SMT > NET (NET only saved 4–34%).
  • 5 lessons learned
    1. Simulation + optimization is the textbook prescriptive pattern: simulate the system, optimise its decision variables.
    2. The model must be calibrated and validated first (R², RMSE, NSE) — otherwise the "optimum" is meaningless.
    3. Stage-specific thresholds (SMT) beat a single constant threshold (NET): timing matters as much as amount.
    4. You can cut a resource ~30–45% without losing output — big win because the objective was water productivity, not yield alone.
    5. Prescriptive analytics turns a scientific model into a farmer-usable decision-support tool.

↔Ch.8R16 · Data-Driven–Simulation–Optimization (DDSO) for Invasive Plants (Zhang 2026)

  • Main argument: Invasion control needs an integrated DDSO framework — data-driven parameters + simulation + optimization — because static strategies fail against a dynamic, spatially spreading invasion under a fixed budget.
  • Problem/Context: Spartina alterniflora is invading China's coastal wetlands — an ecological and economic threat; management money is limited and the invasion changes over space and time.
  • Data & Method: three coupled modules — (1) data-driven ecological parameterization from multi-source observations → time-varying parameters; (2) simulation of life-cycle transitions + spatial dispersal; (3) mixed-integer optimization allocating control budgets & intervention intensities across space and time. Benchmarked against System Dynamics (SD) and plain Simulation–Optimization (SO); scenario analysis over budget levels & invasion intensities.
  • Key results: DDSO beats SD and SO on budget efficiency and yields two counter-intuitive insights: effectiveness depends more on having a coordinated optimization framework than on how much money is spent, and efficient strategies deliberately allocate resources very unevenly across space. Early intervention + cross-regional coordination pay off.
  • 5 lessons learned
    1. The full prescriptive stack — descriptive (data) → predictive (simulation) → prescriptive (optimization) — in one system.
    2. Coordination beats budget: how you allocate matters more than how much you spend.
    3. Optimal spatial allocation is uneven — "fair" equal spreading is economically inefficient.
    4. Mixed-integer programming fits yes/no + how-much decisions across many locations and periods.
    5. Time-varying, data-driven parameters let the plan adapt; static models mis-manage dynamic systems.

↔Ch.8 · MCDMR17 · AHP in Disaster Management (flood & landslide) (Thammaboribal 2025 · AIT)

  • Main argument: AHP, an MCDA method, is an effective, transparent way to combine quantitative data with expert judgment for disaster site-suitability/susceptibility mapping — but methodological variability across studies threatens robustness.
  • Problem/Context: flood and landslide risk mapping needs many criteria of unequal importance; purely data-driven maps ignore expert knowledge, purely expert maps are inconsistent. (Ties to Tutorial 2 / SuperDecisions.)
  • Data & Method: systematic review of ~20 landslide + ~20 flood studies using AHP with GIS. AHP mechanics: build the hierarchy (goal → criteria → sub-criteria → alternatives) → pairwise comparison matrix (Saaty 1–9 scale) → derive weights (eigenvector) → check the Consistency Ratio (CR < 0.1) → weighted overlay in GIS → susceptibility map. Reviews sub-criteria weighting schemes (straight/reciprocal/exponential ranking, rank-order centroid).
  • Key results: Landslide maps are dominated by slope, lithology, land-use/land-cover; flood maps by rainfall intensity, proximity to rivers, drainage density. AHP is not universally applicable — criteria must be context-specific — and the paper calls for a standardised methodology to make studies comparable.
  • 5 lessons learned
    1. AHP = the "few alternatives" optimization row of Ch.8's Table 8.1 — structure a messy multi-criteria decision.
    2. The consistency ratio is what makes expert judgment defensible (CR < 0.1 or redo the comparisons).
    3. AHP's strength = it fuses hard data with expert opinion; its weakness = subjectivity in criteria and weights.
    4. AHP + GIS is the standard pairing for spatial risk decisions (weighted overlay of criteria layers).
    5. Different hazards need different criteria — don't copy a landslide model onto a flood problem.

↔ Expert Systems

↔ESR18 · Rule-Based Expert System for Early Diagnosis of Bipolar & MDD (Zolfagharnasab 2025)

  • Main argument: A rule-based expert system encoding psychiatrists' heuristics can distinguish overlapping mood disorders early — where data-hungry ML struggles and misdiagnosis is costly.
  • Problem/Context: Major Depressive Disorder and Bipolar Disorder share symptoms (sadness, exhaustion), so BD is routinely misdiagnosed as MDD — and the wrong treatment can trigger mania. Early, explainable diagnosis is a community-health need.
  • Data & Method: ontology + SWRL semantic rules written from expert recommendation. Two rule groups: one evaluates single symptoms, one evaluates symptom combinations (eg sadness + mood swings ⇒ evidence of BD). Classes: MDD · manic BD · depressive BD · ordinary/mild. Validated by expert psychiatrists + 120 clinical samples, using only 17 specified symptoms.
  • Key results: classification success rate 93.33%; user-acceptance score 3.56 / 4 — high acceptance from clinicians because the reasoning is transparent.
  • 5 lessons learned
    1. ES suit narrow, judgment-heavy domains where expertise is scarce and rules can be articulated.
    2. Explainability wins clinical trust — a black-box classifier would not be accepted for diagnosis.
    3. Rules can capture combinations of symptoms, not just single facts — that's where expert judgment lives.
    4. ES augment, not replace the psychiatrist (screening/decision support, final call is human).
    5. Knowledge validation with real experts + real cases is the essential final step of knowledge engineering.

↔ES + Ch.6/7R19 · Social4Fashion — Expert System Forecasting Fashion Trends (Balloni 2024)

  • Main argument: A modern "expert system" can automate trend scouting — a deep-learning pipeline over social media replaces designers physically visiting fashion districts.
  • Problem/Context: influencers now shape fashion trends online, and trend forecasting is manual, slow and subjective; designers need faster, evidence-based signals.
  • Data & Method: end-to-end pipeline on Instagram posts collected via hashtags supplied by domain experts (the expert knowledge input): (1) data acquisition → (2) fashion / non-fashion image classification → (3) object detection of the garment (eg handbag: position + class) → (4) dominant colour extraction → (5) trend analysis & forecasting for designers. Validated on real-world service datasets.
  • Key results: fashion-image classification accuracy 97% (95% CI 0.95–1.00); handbag detection mAP 0.77 (CI 0.73–0.82) — good enough for practical fashion-domain analysis; a new dataset is released.
  • 5 lessons learned
    1. "Expert system" today can mean expert knowledge + deep learning (experts supply the hashtags/labels, CNNs do the perception).
    2. Pipeline design matters: classify → detect → extract colour, each stage feeding the next.
    3. Different tasks need different metrics: accuracy for classification, mAP for object detection.
    4. Social media is a live trend sensor — Ch.6 (social analytics) fused with Ch.7 (CNNs).
    5. The business value is speed: automated scouting replaces slow, costly manual trend research.

↔ Big Data Analytics

↔BigDataR20 · Diagnostic Learning Analytics (DLA) in Big Data (Jiang 2025) — your presentation

  • Main argument: A Diagnostic Learning Analytics model built on distributed clustering + pose estimation can mine classroom behaviour data to diagnose learning problems — going beyond static grade data.
  • Problem/Context: traditional learning analytics rely on static data (assignments, transcripts) and miss real behavioural engagement; big-data scale demands distributed processing.
  • Data & Method: M-Canopy-K-means clustering (Canopy pre-clustering to pick initial centroids, parallelised with MapReduce) + PSSHRNet pose-estimation network to recognise classroom behaviour from video; benchmarked on MSCOCO & CrowdPose; applied to real classroom behaviour data.
  • Key results: clustering accuracy +10% vs K-means and +7% vs mini-batch K-means, with >2× speedup on small datasets; PSSHRNet converges faster with lower loss. Real finding: engagement participation score only 2.81, with mobile-phone use = 33.29% of classroom behaviour → targeted personalised intervention.
  • 5 lessons learned
    1. Big-data analytics reaches education: behaviour data diagnoses learning problems that grades hide.
    2. MapReduce parallelisation is what makes clustering feasible at classroom-video scale (Ch. Big Data).
    3. Better centroid initialisation (Canopy) fixes K-means' classic weakness → accuracy + speed.
    4. Combining unsupervised clustering with deep learning (pose estimation) beats either alone.
    5. Analytics must end in action: the diagnosis triggers personalised teaching intervention.

↔BigDataR21 · Integrated Big Data Analytics Architecture for Resilience (Last-Mile Agri-Food) (Alaeddini & Asgari 2025)

  • Main argument: Supply-chain resilience requires an integrated BDA architecture combining descriptive + predictive + prescriptive analytics — not isolated analytics tools.
  • Problem/Context: last-mile delivery (LMD) of agri-food must guarantee uninterrupted access to essential food supplies, but is highly vulnerable to disruption; existing BDA reference models don't address resilience.
  • Data & Method: case-study approach at a large B2B2C last-mile food-delivery company; requirements drawn from industry standards + academic research + company needs. Reviews three literature groups (general BDA reference models eg Kaufmann's; resilient BDA systems eg BDA+IoT for disaster prediction; domain architectures) → designs a layered architecture (ingestion → storage → analytics → decision) carrying all three analytics types.
  • Key results: the proposed architecture enhances resilience by improving anticipation, response and recovery from supply-chain disruptions — descriptive shows what happened, predictive anticipates disruption, prescriptive plans the recovery.
  • 5 lessons learned
    1. Architecture matters: value comes from integrating the analytics layers, not from any single model.
    2. The descriptive → predictive → prescriptive ladder maps directly onto anticipate → respond → recover.
    3. Resilience is a BDA use case in its own right (disruption detection, not just efficiency).
    4. Reference architectures must be adapted to the domain — a generic big-data stack isn't enough.
    5. IoT/sensor streams (velocity) are what make disruption anticipation possible.

↔ Ch.10 / Ch.11 · AI trends, ethics & effective use

↔Ch.10R22 · AI-Enhanced Business Intelligence for Decision-Making (Olszak & Bartuś 2025)

  • Main argument: AI + BI shifts analytics from reactive to proactive/augmented — systems autonomously detect patterns and suggest optimal strategies, replacing intuition and static BI reports.
  • Problem/Context: data explosion + market uncertainty; managers still decide from intuition and rear-view BI dashboards (rational / bounded-rational models). How organisations actually implement AI-enhanced BI is under-studied.
  • Data & Method: qualitative study — in-depth interviews (IDIs) across 20 organisations in three sectors (services, trade, manufacturing) actively using BI + AI; respondents = managers, data analysts, AI/BI specialists. Explores benefits, barriers and perceived synergy.
  • Key results: capabilities = automated insight generation, natural-language querying, forecasting, prescriptive recommendation. Barriers = data quality, skills shortage, trust, governance. Net: AI-enhanced BI is a decision enabler, conditional on organisational readiness.
  • 5 lessons learned
    1. BI's future is augmented analytics — the system proposes, the manager disposes.
    2. The shift is reactive → proactive: from "what happened" to "what to do next", automatically.
    3. Trust and explainability are the gating factor for managerial adoption, not model accuracy.
    4. Classic barriers persist: data quality, skills, governance (same list as Big Data's CSFs).
    5. BI is a management paradigm, not a technology — AI amplifies it but doesn't replace the organisation.

↔Ch.10R23 · Generative AI for Autonomous Data Analytics (Fikardos 2026)

  • Main argument: Exposing an analytics platform's APIs as tools callable by LLM agents lets analysts run whole analytical workflows in natural language — a step toward autonomous data analytics.
  • Problem/Context: form/API-based analytics platforms impose cognitive overhead and technical barriers; non-experts can't drive them, so insight is bottlenecked on specialists.
  • Data & Method: convert platform APIs into LLM function-calling tools; a coordination process links analyst ↔ agents ↔ platform; result = a chat interface for analytics. Instantiated with open-source models. Evaluated by a user study (chat vs form-based UI) on task time, number of actions, usability, satisfaction, confidence, hallucination.
  • Key results: task time 6.71 → 2.70 min; actions 14.48 → 3.30; usability 3.09 → 4.64; +7.2% mean overall score vs baselines; chat better by ≥1 point on usability, ease of use, satisfaction, timeliness; notably lowers the "familiarity barrier" (users need not know the platform or use case).
  • 5 lessons learned
    1. Natural language is the next BI interface — it collapses the tool-learning cost (Ch.9 tooling).
    2. Function calling / tool use is what turns a chatbot into an agent that actually executes analytics.
    3. The measurable win is time and effort (2.5× faster, 4× fewer actions), not just user delight.
    4. GenAI democratises analytics — non-experts run workflows previously reserved for specialists.
    5. Risks remain: hallucination and trust must be measured explicitly, not assumed away.

↔Ch.11R24 · Big Data Analytics Democratized: Clean Rooms & Customer Privacy Choice (Pauwels & Aksehirli 2025)

  • Main argument: Omnichannel data creates three challenges — organizational (silos), operational (generating insight), societal (privacy) — met by three solutions: democratize digital transformation · data clean rooms · customer privacy choice by design.
  • Problem/Context: customer data is trapped in competing "walled gardens" (retail-media platforms), attribution across channels is broken, and privacy regulation + consumer distrust are rising. Analysts still spend >80% of effort on the data part.
  • Data & Method: conceptual/viewpoint paper (marketing-strategy lens) synthesising industry practice and literature; frames three challenge levels and proposes three matched solution frameworks — within-business leadership, cross-business data trade (clean rooms), and customer privacy choice — plus a research agenda.
  • Key results: (1) Democratization — spread analytics skills & leadership across the business, breaking silos. (2) Data clean rooms — a neutral space where competitors/platforms collaborate on data without exposing raw customer records. (3) Privacy choice — let customers choose their data-sharing level, turning compliance into trust.
  • 5 lessons learned
    1. The hardest big-data problems are organizational and societal, not technical.
    2. Walled gardens break attribution — clean rooms are the privacy-preserving workaround.
    3. Democratizing analytics (skills + leadership) matters more than buying another tool.
    4. Privacy by choice/design turns a legal burden into a competitive trust advantage.
    5. Ethics is now a core analytics design constraint (Ch.11), not an afterthought.

↔Ch.1/11R25 · Antecedents to the Effective Use of BI — Systematic Review (Widjaja 2025)

  • Main argument: Research obsesses over BI adoption, but performance comes from effective USE — and the antecedents of effective use are technological, organizational, environmental and human.
  • Problem/Context: BI is directly linked to performance, yet studies of its effective use are scarce; BI is "not only about novel technology but about the human user who might or might not take advantage of it."
  • Data & Method: systematic literature review (PRISMA-style) covering two decades; a very large initial hit set (~23k) screened down to quantitative studies of the antecedents of actual BI use (not intention). Iterative coding produced 53 factors synthesised into 10 categories.
  • Key results: 10 categories — BI system capabilities, technology, data-related factors, organizational factors, HR management, leadership, environmental factors, individual behavioral, individual capabilities (+ infrastructure). Related concepts: BI infusion and embeddedness of use; effective use is measured by intensity and extent of use.
  • 5 lessons learned
    1. Adoption ≠ effective use — buying BI doesn't mean anyone uses it well; only use drives performance.
    2. Antecedents are socio-technical: system/data quality AND leadership, HR, individual capability.
    3. Leadership and culture recur as decisive (echoing R2 and the Big Data CSFs).
    4. Effective use is measurable — via intensity & extent of use, infusion, embeddedness.
    5. Research (and managers) should shift focus from acquiring BI to embedding it in daily decisions.

Formulas, Metrics & Quick-Pick

Classification — Accuracy=(TP+TN)/all · Precision=TP/(TP+FP) · Recall/TPR/Sensitivity=TP/(TP+FN) · Specificity=TN/(TN+FP) · F1=2PR/(P+R) · AUROC 0.5=chance, 1=perfect (robust to imbalance) · Cohen's κ = agreement beyond chance.

Regression — MAE (mean |error|) · MSE · RMSE=√MSE (same units as y; penalises big errors) · R² = % variance explained (↑ better). NSE (Nash–Sutcliffe, R15) ↑ better.

Clustering — Silhouette ↑ better · Davies-Bouldin ↓ better · elbow method for k. Detection — mAP (mean Average Precision, R19).

LP — Max/Min Z = Σcᵢxᵢ s.t. Σaᵢⱼxⱼ ≤ bᵢ, xᵢ ≥ 0. Expected value = Σ pᵢ × payoffᵢ. AHP — Saaty 1–9 pairwise scale → eigenvector weights → CR < 0.1.

Which technique? — images → CNN (+ ensemble/soft voting) · sequences/text-in-time → RNN/LSTM · language understanding → BERT/LLM · numeric target → regression (RF/XGBoost beat linear) · scarce data + expert knowledge + need to explain → expert system (rules) · best decision under constraints → LP / mixed-integer optimization · too complex to optimise → simulation · many criteria + expert judgment → AHP · huge/streaming/unstructured data → Hadoop / MapReduce / stream analytics.
