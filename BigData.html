Key idea

Core message: "Big Data" is a misnomer — it is more than just big. It is data whose volume, variety, and velocity (plus veracity and variability) exceed what traditional platforms can capture, store, and analyse. And crucially: Big Data by itself is worthless — Big Data + "big" analytics = value. The technology answer is a new stack (MapReduce, Hadoop, NoSQL, stream analytics) that coexists with, rather than replaces, the data warehouse.

Opening vignette — telecom churn with Big Data methods

A telecom company (highly competitive market, churn rate higher than most industries) analyses customer churn with Big Data methods. The challenges that made this a Big Data problem rather than a classic data-mining one:

  • Data from multiple sources (call records, web logs, customer care, network events) that must be combined;
  • Data volume far higher than usual — too large for the traditional platform.
Key vocabulary — sessionizing: stitching raw event/click records into coherent user sessions so behaviour can be analysed end-to-end. The vignette's discussion questions hinge on why sessionizing the multi-source event data was necessary before churn modelling.

Definition, sources, and just how big "big" is

Big Data means different things depending on background; traditionally it meant massive volumes (CERN, NASA, Google). Where does it come from? Everywhere: web logs, RFID, GPS, sensor networks, social networks, Internet text and search indexes, call detail records, astronomy/biology/genomics/physics experiments, medical records, military surveillance, multimedia archives…

Fundamentals — and when you actually need Big Data tech

Big Data brought big challenges: effectively and efficiently capturing, storing, and analysing the data — needing a new breed of technologies (developed, purchased, hired, or outsourced).

"Big Data considerations" — signals your current platform is not enough

  • You can't process the amount of data you want to on the current platform.
  • You can't include new/contemporary sources (social media, RFID, sensory, web, GPS, text) because they don't fit the storage schema.
  • You need to integrate data as quickly as possible to keep analysis current.
  • You want schema-on-demand storage because of data variety.
  • Data arrives too fast for the traditional analytics platform to handle.

Critical success factors for Big Data analytics (7 — memorise)

  1. A clear business need — alignment with the vision and strategy (business value, not tech fashion).
  2. Strong, committed sponsorship — an executive champion.
  3. Alignment between the business and IT strategy.
  4. A fact-based decision-making culture.
  5. A strong data infrastructure.
  6. The right analytics tools.
  7. Right people with right skills (personnel with advanced analytical skills).
Exam link: these CSFs parallel the BI success factors from R2/R25 (sponsorship, alignment, culture) — a nice cross-reference if a question asks "what makes Big Data initiatives succeed?"

Enablers & challenges

Enablers (performance technologies)

  • In-memory analytics — store and process the complete dataset in RAM.
  • In-database analytics — place analytic procedures close to where the data lives.
  • Grid computing & MPP — many machines/processors in parallel (massively parallel processing).
  • Appliances — hardware + software + storage combined in one unit for performance and scalability.

Challenges

  • Data volume — capture, store, process in a timely manner.
  • Data integration — combine quickly and at reasonable cost.
  • Processing capabilities — process data as it is captured (stream analytics).
  • Data governance — security, privacy, access.
  • Skill availability — the data scientist shortage.
  • Solution cost — proving ROI.

Business problems addressed by Big Data analytics

Process efficiency and cost reduction · brand management · revenue maximisation (cross-/up-selling) · enhanced customer experience · churn identification & customer recruiting · improved customer service · identifying new products and market opportunities · risk management · regulatory compliance · enhanced security capabilities.

MapReduce

What it is: a programming model (developed and popularised by Google) that distributes the processing of very large multi-structured files across a large cluster of ordinary ("simple") machines — achieving high performance from commodity hardware.

  • Map step: split the data into parts, process each part in parallel (produce key–value pairs).
  • Reduce step: collect, aggregate, and combine the intermediate results into the final answer.
  • Good at: indexing the web for search, graph analysis, text analysis, machine learning over huge multi-structured data.
Keep straight for the exam: MapReduce provides the control/distribution framework for analytics — it is not the analytics itself (one of the "demystifying facts" below).

Hadoop & its ecosystem

Hadoop = an open-source framework for storing and analysing massive amounts of distributed, unstructured data. Originally created by Doug Cutting at Yahoo!, now an Apache Software Foundation project. Clusters run on inexpensive commodity hardware, so projects scale out cheaply.

How Hadoop works

Ingest unstructured / semi-structured data→Break into "parts"→Load into HDFS across nodes→Replicate each part (failsafe)→Distribute jobs to nodes→Collect & aggregate results via MapReduce

Ecosystem sub-projects (recognise the names)

Demystifying facts (Technology Insight 2 — quotable list)

  • Hadoop consists of multiple products — it's an ecosystem, not a single product.
  • Open source, but also available from vendors.
  • HDFS is a file system, not a DBMS.
  • Hive resembles SQL but is not standard SQL.
  • Hadoop and MapReduce are related but not the same.
  • MapReduce provides control for analytics, not analytics itself.
  • Hadoop is about data diversity, not just data volume.

Big Data and the data warehouse — coexistence, not replacement

Big Data and RDBMS "do not go nicely together" — but Hadoop will not replace the data warehouse; each has use cases (Hadoop = repository & refinery, active archive; DW = performance, integrated business-value data, interactive BI tools).

Coexistence patterns (5)

  1. Hadoop stores and archives multi-structured data.
  2. Hadoop filters, transforms, consolidates multi-structured data (the "refinery").
  3. Hadoop analyses large volumes and publishes results (to the DW).
  4. A relational DBMS with MapReduce capabilities serves as an investigative computing platform.
  5. A front-end query tool accesses and analyses data across both.

Vendors & platforms

A fast-developing landscape: Cloudera, MapR, Hortonworks (the classic Hadoop distributors) — plus IBM (Netezza, InfoSphere BigInsights), Oracle (Exadata/Exalogic), Microsoft, Amazon, Google. Platform example in the slides: Teradata Aster (SQL + MapReduce analytics on one platform).

Big Data and stream analytics

Stream analytics = data-in-motion / real-time analytics: the analytic process of extracting actionable information from continuously flowing data — the Velocity V made operational.

  • Why: it may not be feasible to store all the data first, or the data would lose its value if analysed later (fraud alerts, outage detection, dynamic pricing).
  • Stream vs perpetual analytics: stream = act on a window as it flows; perpetual = continuously re-evaluate everything.
  • Critical event processing: detect and react to meaningful event patterns in the stream.
Energy-industry use case (slides' worked figure): sensor data from the production system + meteorological data (wind, light, temperature) + usage data from smart meters/grid devices → data integration & temporary staging → streaming analytics predicting usage, production, and anomalies → feeds capacity decisions (production side) and pricing decisions (consumption side), with permanent storage alongside.

Application sectors: e-commerce, telecommunications, law enforcement & cyber security, power industry, financial services, health services, government.
